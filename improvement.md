# Parser Improvement Plan

This document outlines the detailed diagnosis, root causes, and a step-by-step action plan to address the 10 problematic premise types and AST structures identified in the parser output.

---

## 1. Main Diagnosis: The `RULE` Kind is Too Broad

The core architectural issue is that the `RULE` premise kind is currently treated as a catch-all for any statement containing conditional or universal properties. This conflates simple conditional rules with:
*   **Nested Implications / Meta Rules:** `(A(x) -> B(x)) -> C(x)`
*   **Mixed Quantifier Scopes:** `exists x A(x) -> forall y B(y)`
*   **Deontic Obligations:** `mandatory(A(x))`
*   **Universal Facts:** `forall x A(x)`

Furthermore, because the Pydantic schema enforces strict AST shapes during deserialization, any malformed AST generated by the LLM raises a hard `ValidationError` and crashes the compilation phase before the **Stage 4 Validator & Stage 5 Repair Loop** can run.

---

## 2. Problem-by-Problem Breakdown & Action Items

| Priority | Problem Type | Root Cause | Proposed Fix |
| :--- | :--- | :--- | :--- |
| **1** | **META / nested rule** | Stage 1 classifies nested implications as simple `RULE` kinds, causing Stage 3 to collapse or flatten them. | Update Stage 1 classification to detect nested logic and classify as `META`. Add few-shot RAG examples of nested implications. |
| **2** | **RULE-about-RULE** | The parser collapses implication-in-implication structures into flat `and`/`not` trees. | Enforce nested `implies` nodes in Stage 3. Add validator check in Stage 4 to verify implication nestedness. |
| **3** | **EXISTS mixed with FORALL** | Quantifiers are flattened into a single scope, losing independent variable binding. | Instruct Stage 3 to nestedly scope quantifiers: an outer `implies` with left-nested `exists` and right-nested `forall`. |
| **4** | **EXISTS mislabeled as RULE** | Triggers like "There exists at least one..." are misclassified as `RULE` when part of longer sentences. | Refine Stage 1's existential heuristics and classification system prompt. |
| **5** | **FORALL mislabeled as RULE** | Universal statements ("All X do Y") without explicit conditional words are classified as `RULE`. | Classify "All X do Y" as `FORALL` kind in Stage 1 when there is no conditional connector. |
| **6** | **Malformed quantifier AST** | Stage 3 LLM violates children counts, causing Pydantic validation to crash the pipeline. | **Relax Pydantic schema validation**; move strict shape checks to Stage 4 Validator so Stage 5 can repair them. |
| **7** | **Malformed implication AST** | LLM outputs `implies` with `< 2` children, leading to hard Pydantic validation crashes. | Move implication child count verification to Stage 4 Validator and leverage Stage 5 Repair. |
| **8** | **Atomic predicate missing args** | LLM generates `predicate()` instead of `predicate(x)`. | Move arity/argument check to Stage 4 Validator and repair. |
| **9** | **Obligation/Deontic collapsed** | Deontic statements ("It is mandatory...") lose their obligation semantics. | Introduce an `OBLIGATION_RULE` kind / deontic wrapper or property in Stage 3 AST. |
| **10** | **Compound antecedent split** | Attributes of the subject are incorrectly split between the antecedent and consequent. | Refine Stage 1 splitting instructions to keep all subject qualifiers in the `if_part`. |

## 3. Step-by-Step Implementation Plan

### Step 1: Relax Pydantic Schema Validation (Allowing Repair)
Currently, [schemas.py](file:///c:/Users/hi/Desktop/Praser/logic_pipeline/src/schemas.py) throws hard `ValueError` exceptions inside the `@model_validator(mode="after")` decorator. We will modify the model validator to log warnings or allow construction, moving the strict error checks to the `LogicValidator`.

> [!IMPORTANT]
> If Pydantic crashes during `Stage3Output.model_validate()`, the pipeline terminates. Moving validations to `LogicValidator` allows the `RepairLoop` to catch the issues and ask the LLM to fix them.

```diff
# logic_pipeline/src/schemas.py
     @model_validator(mode="after")
     def check_shape(self):
-        if self.type == "atomic":
-            if not self.name:
-                raise ValueError("atomic node requires name")
-            if not self.arguments:
-                raise ValueError("atomic node requires at least one argument")
+        # Allow parsing of potentially invalid shapes so the validator and repair loop can handle them
+        return self
```

### Step 2: Update Stage 4 Validator (`stage4_validate.py`)
Transfer the strict AST shape checks from the schema validator to [stage4_validate.py](file:///c:/Users/hi/Desktop/Praser/logic_pipeline/src/stage4_validate.py):

1.  **Quantifier Children:** Verify that `forall` and `exists` nodes have exactly 1 child.
2.  **Implication Children:** Verify that `implies` nodes have exactly 2 children.
3.  **Atomic Arguments:** Verify that `atomic` nodes have `len(arguments) >= 1`.
4.  **Nested Rule Detector:** If a `RULE` contains an `implies` node nested inside another `implies` node, raise a warning/error suggesting reclassification to `META`.

---

### Step 3: Enhance Stage 1 CNL Heuristics (`stage1_cnl.py`)
Modify the system prompt in [stage1_cnl.py](file:///c:/Users/hi/Desktop/Praser/logic_pipeline/src/stage1_cnl.py) to improve classification accuracy:

*   **Existential vs. Rule Heuristic:** If a sentence asserts existence ("there exists", "at least one", "some"), classify it as `EXISTS` unless it is a conditional premise.
*   **Universal Heuristic:** If a sentence asserts a property of all members ("all", "every", "each") without conditional words, classify it as `FORALL`.
*   **Meta / Nested Rule Detector:**
    *   Explicitly check if the statement contains nested conditionals (e.g., "If A implies B, then C").
    *   Classify as `META` and add the `nested_logic` or `meta_statement` risk flags.
*   **Deontic Obligation Heuristic:** Ensure words like "mandatory", "obligatory", "required to" trigger `OBLIGATION_RULE` and the `deontic_obligation` flag.
*   **Compound Antecedent Splitting Instructions:** Update instructions for extracting `if_part` and `then_part`:
    > Keep all qualifying attributes of the subject (e.g. "third-year student who gets an internship") in the `if_part`.

3.  **Strict Shape instructions:** Re-emphasize that `forall`/`exists` must have exactly 1 child, `implies` must have 2 children, and `atomic` must have at least 1 argument.
